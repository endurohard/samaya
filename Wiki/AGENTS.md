# Wiki Schema (Karpathy LLM Wiki pattern)

Эта вики — **вторичный слой знаний** о проекте samaya, который ведёт LLM-агент. Первичные источники (код, миграции, docs) никогда не модифицируются этой вики.

## Три слоя

1. **Raw sources** (read-only для вики): код проекта, миграции БД, `docs/*.md`, `*.md` в корне, `docker-compose*.yml`, конфиги.
2. **Wiki** (этот каталог): `.md` файлы, которые агент пишет и поддерживает. Любой факт здесь имеет ссылку на raw source.
3. **Schema** (этот файл): правила ingest/query/lint.

## Структура каталогов

```
Wiki/
├── AGENTS.md          — схема (этот файл)
├── index.md           — каталог всех страниц
├── log.md             — append-only хронология ingest/query/lint операций
├── sources.md         — карта raw-sources → страницы вики
├── services/          — одна страница на сервис/модуль
├── concepts/          — кросс-сервисные концепты (auth, events, и т.п.)
└── decisions/         — ADR-подобные записи решений
```

## Операции

### Ingest
Когда появляется новый источник (фича, миграция, документ, сервис):
1. Прочитать источник.
2. Определить, какие entity/concept страницы затрагивает (обычно 3–10).
3. Обновить каждую: добавить факт, поставить `[[cross-link]]` на связанные страницы.
4. Обновить `index.md` если появилась новая страница.
5. Добавить строку в `log.md`: `## [YYYY-MM-DD] ingest | <источник> → <страницы>`.

### Query
Когда пользователь задаёт вопрос:
1. Сначала искать ответ в вики (навигация через `index.md` и `[[links]]`).
2. Если нашёл — ответить + проверить свежесть против raw source.
3. Если не нашёл — прочитать raw source, ответить, **подшить результат в подходящую страницу** (это ключевой шаг: query компаундит знания).
4. Залогировать в `log.md`: `## [YYYY-MM-DD] query | <вопрос> → <обновлённые страницы>`.

### Lint
Периодически (по запросу пользователя):
- Сироты: страницы без входящих ссылок.
- Противоречия: факты в разных страницах, которые не сходятся.
- Устаревшее: ссылки на файлы/функции, которых больше нет в repo.
- Пропуски: raw-sources без покрытия в вики (через `sources.md`).

## Конвенции страниц

- **Frontmatter**: `type`, `status` (`stable`/`wip`/`stale`), `last_verified` (дата), `sources` (список путей в repo).
- **Ссылки**: Obsidian-style `[[services/some-service]]`.
- **Факты**: каждый нетривиальный факт должен цитировать путь в repo, например: `JWT секрет читается из env (services/user-service/src/config/index.ts:12)`.
- **Не дублировать код**: ссылка > копирование. Вики объясняет *почему*, код показывает *как*.
- **Язык**: русский для объяснений, английский для имён сущностей/кода.

## Что НЕ писать в вики

- Копии кода, которые устареют через неделю.
- Пошаговые туториалы — они живут в `docs/`.
- Временный контекст задачи (это для планов/тасков, не для вики).
- Факты, которые тривиально выводятся из `git log` или текущего состояния файла.

## Запуск

Пользователь работает в Obsidian (vault = эта папка), агент (Claude Code) правит `.md`. Пользователь видит изменения в реальном времени через граф-вью и backlinks.
