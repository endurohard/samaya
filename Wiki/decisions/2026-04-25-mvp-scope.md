# ADR-001: MVP Scope для samaya

**Дата:** 2026-04-25
**Статус:** принято (предварительно, ждёт подтверждения владельца)

## Контекст

Цель — сделать SaaS/in-house платформу для бьюти-салона Samaya, повторяющую ключевую функциональность dikidi.ru. Полный функционал dikidi (см. [[../concepts/dikidi-feature-map]]) — это 11 модулей, ~50 экранов, ~3–6 месяцев разработки.

Стек выбран: Node.js/TypeScript + Express, PostgreSQL, Redis, Kong (DB-less), всё в `docker-compose` — повторяет [[~/work/food-flow]].

## Решение: Phase 0 = booking-only MVP

Делаем минимальный срез, который уже даёт ценность: владелец видит расписание, клиент может записаться онлайн. Остальное — пострелизные фазы.

### В Phase 0 включено

| Модуль dikidi | Что делаем |
|---|---|
| Журнал записей | Календарь по мастерам день/неделя, CRUD записей, статусы |
| График работы | Расписание мастеров (working hours per day) |
| Клиенты | Список клиентов + сегменты (постоянные/спящие/пропавшие/новые/без записей/заблокированные/удалённые), поиск, CRUD, экспорт CSV. Расширено в [[2026-04-25-clients-segments-in-mvp]] |
| Настройки (минимум) | Компания, услуги, мастера, роли |
| Публичный виджет | Страница компании + форма записи (выбор услуги → мастера → даты → времени → клиент) |

### Сервисы Phase 0

| Сервис | Порт | Ответственность |
|---|---|---|
| `user-service` | 3001 | Auth (JWT + refresh), пользователи, роли (`owner`/`master`/`admin`/`client`) |
| `salon-service` | 3002 | Компания, услуги (категория/цена/длительность), мастера, расписание мастеров |
| `booking-service` | 3003 | Слоты, записи, lifecycle (`pending`/`confirmed`/`completed`/`canceled`/`no_show`) |
| `client-service` | 3004 | Клиенты салона, история визитов, заметки |

Frontend в Phase 0 — отдельный `frontend-service` (статика):
- `public/` — виджет онлайн-записи (одностраничник).
- `admin/` — журнал записей + график + клиенты.

### Отложено (Phase 1+)

| Модуль | Фаза | Причина отложить |
|---|---|---|
| Финансы (доходы/расходы/счета/кассы) | 1 | Завязан на продажи; без них не имеет смысла |
| Продажи (чеки) | 1 | Расширение записи, требует кассу |
| Кассовые операции (54-ФЗ) | 2 | Внешняя интеграция, нужен ОФД |
| Эквайринг | 2 | Внешняя интеграция |
| Зарплата | 2 | Считается на основе продаж + графика, оба должны быть стабильны |
| Товары/склад | 2 | Параллельный домен |
| Рассылки (WhatsApp) | 1 | WA Business API setup, ценно сразу для retention |
| Бонусы/сертификаты/чаевые | 2 | CRM-расширение (бонусный баланс уже есть в схеме clients) |
| RFM-аналитика «Возвращаемость» | 1 | Поверх существующих сегментов |
| Отзывы | 2 | После запуска |
| Звонки/телефония | 3 | Опционально |
| Мобильные апки | 3 | После веба |

## Архитектурные решения, фиксируемые сразу

Учтены косяки food-flow (см. food-flow `decisions/2026-04-11-deep-audit`):

1. **JWT secret**: fail-hard в `NODE_ENV=production`, никаких hardcoded fallback'ов. Один общий способ чтения секрета через shared-package.
2. **Auth-middleware с первого дня** во всех сервисах. Никаких "забыли auth на endpoint".
3. **Multi-tenant с первого дня**: `company_id` в JWT-claim'е, во всех таблицах, во всех SQL-запросах. Если samaya — single-tenant, всё равно делаем multi-tenant-ready (1 компания), чтобы не переписывать.
4. **Race conditions**: бронирование слота — ВСЕГДА через `SELECT ... FOR UPDATE` + транзакция. Два клиента не должны забронировать один слот.
5. **Тесты**: с первого дня jest/vitest + 1 интеграционный тест на сервис как минимум. Не повторяем "0 тестовых файлов при `"test": "jest"`".
6. **Outbox pattern для событий**: если используем RabbitMQ/Redis pub-sub, события публикуем через outbox-таблицу, не напрямую (чтобы не терять события при сбое брокера).
7. **БД**: один Postgres, схемы по сервисам (`users`, `salons`, `bookings`, `clients`). Проще для MVP, переход на DB-per-service возможен позже.

## Открытые вопросы (требуют ответа владельца до старта кодинга)

1. **Single-tenant (только Samaya) или SaaS на много салонов?** — определяет, нужна ли регистрация компаний и тарифный план.
2. **Использовать ли реально WhatsApp Business API в Phase 1**, или fallback на Telegram-бота (как в food-flow есть)?
3. **Дизайн-система** — копируем dikidi (тёмная сине-фиолетовая) или своя?
4. **Хост / дев-окружение** — где будем деплоить? VPS, облако, k8s?
5. **Языки UI** — только русский, или ещё английский (для иностранных клиентов косметологии)?
