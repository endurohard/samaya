# ADR-002: Inventory + tech-cards в Phase 0

**Дата:** 2026-04-25
**Статус:** предложено, ждёт подтверждения владельца

## Контекст

В ADR-001 ([[2026-04-25-mvp-scope]]) модуль "Товары/склад" был отложен в Phase 2. Владелец явно потребовал в Phase 0:

> «нужно настроить поступление расходников на склад и добавлять расход к услугам, чтоб сверять остатки продукции»

Концептуальная модель — [[../concepts/inventory-tech-cards]].

Для косметологии это критично: гель, расходники для аппаратов, иглы. Без техкарт невозможно посчитать маржу услуги и провести инвентаризацию.

## Решение (предлагается)

**Бамп `inventory-service` + tech-cards в Phase 0.** Phase 0 теперь = **booking + inventory core**.

### Обновлённый состав сервисов Phase 0

| Сервис | Порт | Phase 0 включает |
|---|---|---|
| `user-service` | 3001 | Auth, пользователи, роли |
| `salon-service` | 3002 | Компания, услуги (с `tech_card_id` reference), мастера, расписание |
| `booking-service` | 3003 | Слоты, записи, lifecycle, **публикация `booking.completed` через outbox** |
| `client-service` | 3004 | Клиенты, история визитов |
| **`inventory-service`** | **3005** | **products, stock_lots, stock_movements, tech_cards, warehouses, supplier_invoices, inventory_counts** |
| `frontend-service` | — | публичный виджет + админка (включая UI склада) |

Поставщики (контрагенты) пока живут в `inventory-service` — позже могут уехать в финансы, но в Phase 0 это утяжеление, делаем locally.

### Что в Phase 0 НЕ включено (остаётся в +1)

Из inventory:
- Перевод между складами (transfers)
- Множественные склады (Phase 0 — один склад на компанию)
- Инвентаризация (counts) — UI, но логика adjustments через manual movements уже работает.
- FEFO (срок годности) — поле `expires_at` в схеме есть, но списание в Phase 0 по чистому FIFO. FEFO поверх — позже.

### Альтернатива (если предложение не примут)

**Опция B** — оставить inventory в Phase 1, но в Phase 0 спроектировать:
- В таблице `services`: nullable `tech_card_id` с самого начала.
- В `bookings`: поле `completed_at` и таблица `booking_completion_events` как outbox-заготовка.
- При complete пока ничего не списывается, но событие в outbox пишется.

Тогда подключение inventory в Phase 1 = миграции inventory-service + воркер на outbox, без переделки уже работающих сервисов.

## Обоснование «бампа в Phase 0»

ЗА:
- Это **главное операционное требование владельца** (после самой записи). Без него платформа не закрывает реальный кейс.
- Откладывание = переписывание `booking-service` (добавление публикации событий), `salon-service` (tech_cards), миграции данных. Дешевле сразу.
- Архитектурно outbox в `booking-service` всё равно нужен для будущих интеграций (нотификации, аналитика) — заложить с первого дня.

ПРОТИВ:
- Phase 0 удлиняется примерно на **1.5–2 недели** работы (inventory-service + UI склада + supplier-invoice форма + выгрузки).
- Усложняется тестирование (FIFO под concurrency требует интеграционных тестов с реальной БД).
- Если первые недели бизнесу нужно ТОЛЬКО онлайн-расписание (booking), он не получит value быстрее.

## Срез "что показать первым"

Если владельцу нужно «как можно скорее увидеть рабочую запись», предлагаю **двухэтапную Phase 0**:

- **Phase 0a** (~1.5 нед): user + salon + booking + client + frontend (виджет + базовый журнал). Без inventory. Внутри `booking-service` сразу outbox-таблица и публикация `booking.completed` (без подписчика пока).
- **Phase 0b** (~1.5–2 нед): inventory-service + UI склада + tech-cards + воркер-подписчик на `booking.completed`. К концу Phase 0b остатки реально считаются.

Так value поступает раньше, и архитектура не страдает (outbox с первого дня).

## Ждём от владельца

1. **Phase 0 = booking+inventory сразу** или **двухэтапная (0a → 0b)**?
2. **Один склад на компанию** в Phase 0 — норм? Или сразу мульти-склад (мастера со своими подскладами)?
3. **Что делать при недостаче расходника** при попытке complete booking-а:
   - блокировать complete (хардкор, но точные остатки),
   - разрешить минусовой остаток с флагом (мягко, остатки могут расходиться),
   - списать сколько есть и предупредить (компромисс)?
