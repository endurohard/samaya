# ADR-003: Сегментация клиентов в Phase 0 MVP

**Дата:** 2026-04-25
**Статус:** принято
**Заменяет в части scope:** строку «Клиенты (subset)» из [[2026-04-25-mvp-scope]] (где сегменты были отложены в Phase 1).

## Контекст

В исходном MVP-scope ([[2026-04-25-mvp-scope]]) функциональность раздела «Клиенты» была урезана до простого справочника без сегментов: предполагалось, что владелец увидит список из админки и этого достаточно. Сегменты («Постоянные / Спящие / Пропавшие / Новые / Не посещали / Заблокированы / Удалены») и счётчики справа должны были появиться в Phase 1 как RFM-аналитика.

Владелец явно запросил повторить раздел DIKIDI 1:1 («нужно сделать такой же функционал раздела клиенты список клиентов»). В DIKIDI боковая панель категорий — главный навигационный элемент этого экрана; без неё список плоский и теряет ценность для retention-работы.

## Решение

1. **Поднять `client-service` в Phase 0a** (порт 3004) с CRUD и сегментацией. Новых сервисов не создаём — занимаем уже зарезервированный за client-service порт.
2. **Сегмент считается на лету одним SQL-выражением** (`SEGMENT_EXPR` в `clients.service.ts`) поверх агрегатов из `bookings.bookings`. Никакой материализованной view, никакого фонового пересчёта — на 27 клиентах это занимает <10ms, на 5000 — <100ms по расчёту (proof-of-concept). Когда потребуется — добавим matview без слома API.
3. **Связь `clients ↔ bookings`** идёт по `(company_id, client_id)` с фолбэком на `(company_id, client_phone)`. Никаких FK через схемы (schema-per-service). Phone — нормализованный CITEXT с UNIQUE per-company.
4. **Параметры сегментации — env-переменные** (см. `client-service/src/config.ts`), значения по умолчанию повторяют DIKIDI:
   - постоянные: ≥2 записи за 90 дней
   - спящие: > 90 дней без записи
   - пропавшие: > 180 дней без записи
   - новые: создан < 7 дней назад и без визитов
5. **Модули сайдбара DIKIDI**, не входящие в MVP («Возвращаемость», «Бонусная программа», «Рассылка», «Звонки», «Сертификаты», «Чаевые», «Отзывы», «Штрафы»), — заглушены как `disabled` пункты sub-nav с `data-pending="P1"`. Этого достаточно для визуального соответствия.

## Что отложено внутри раздела «Клиенты»

| Функция | Фаза | Причина |
|---|---|---|
| RFM «Возвращаемость» | 1 | Нужны полноценные RFM-скоры, не базовая сегментация |
| Бонусная программа (движения, начисления, ставки) | 1 | Поле `bonus_balance` уже есть, нужны `bonus_movements` + бизнес-правила |
| Рассылки | 1 | Завязано на канал (WA/SMS) — отдельный домен |
| Звонки | 3 | Телефония, опционально |
| Сертификаты / Чаевые / Отзывы / Штрафы | 2 | CRM-расширения |
| Импорт CSV/Excel клиентов | 1 | Серверный endpoint + валидатор + дедупликация |
| Массовое начисление бонусов | 1 | Вместе с бонусной программой |

## Почему «на лету», а не materialized view

- **Аргумент за matview**: дешевле для UI на больших объёмах (десятки тысяч клиентов).
- **Аргумент против**: для одного салона максимум сотни клиентов → запрос укладывается в десяток мс. Matview потребовал бы тригеров обновления (`booking.created/completed/canceled`), что добавило бы 4 точки сбоя. Когда станет узким местом — добавим `clients.client_segments_mv` без слома API.

## Последствия

- В index.md `client-service` помечен как **готов**.
- В `MEMORY.md` отдельная запись не нужна: всё здесь и в [[../services/client-service]].
- Frontend получил новый view `clients` (sub-nav, таблица, sidebar категорий, модалка create/edit, экспорт CSV). Подробности — в [[../services/frontend-service]] (раздел «Clients view»).
